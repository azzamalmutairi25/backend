# منصّة كفاءات (Kafaat) — التوثيق المعماري

> منصّة حكومية لتقييم الكفاءات القيادية: تُدير رحلة المرشّح من الإضافة إلى التقرير النهائي المُعتمَد،
> مع ضبط صلاحيات دقيق، تصنيف أمني للبيانات، تشفير للمعلومات الشخصية، وسجلّ تدقيق كامل.

آخر تحديث لهذا الملف يواكب إصلاحات مراجعة الصحّة/الأمان (72 اختباراً، كلها ناجحة).

---

## 1. نظرة عامة

**كفاءات** نظامٌ لمركز تقييم قيادات: يُضاف المرشّح، يُعتمد، يُجدوَل، يحضر جلساته، يُقيَّم على كفاءات محدّدة عبر أنشطة (مقابلة/حلقة نقاش/أدوات قياس/تمرين تكاملي)، ثمّ يُكتب له تقرير نهائي يُعتمد من إدارة التطوير. الشخص الواحد قد يمرّ بعدّة **دورات تقييم** عبر الزمن، ولكلّ دورة رمزٌ فريد وحالة مستقلّة.

المبادئ الحاكمة:
- **سرّية بالتصميم:** الأسماء وأرقام الهوية مشفّرة في القاعدة؛ التصنيف الأمني (عادي/سرّي/سرّي للغاية) يحجب المرشّحين المصنّفين عمّن لا يملك التصريح — بردّ «غير موجود» لمنع كشف الوجود.
- **فصل الأدوار:** إدخال التقييم، اعتماده، كتابة التقرير، واعتماده النهائي — أدوارٌ مختلفة.
- **أثرٌ لكلّ فعل:** كل إجراء حسّاس يُكتب في سجلّ التدقيق مع الفاعل والوقت وعنوان الـIP.
- **نزاهة على مستوى القاعدة:** فهارس فريدة تحسم السباقات المتزامنة بدل الاعتماد على فحوصٍ في التطبيق.

## 2. المنصّة التقنية

| الطبقة | التقنية |
|---|---|
| الخلفية (Backend) | Laravel (PHP) — `backend-new/` |
| الواجهة (Frontend) | Vue 3 + Vite + Vue Router — `frontend/` |
| قاعدة البيانات | PostgreSQL |
| المصادقة | Laravel Sanctum (رموز) + Active Directory اختياري للمستخدمين الداخليين |
| التشفير | Laravel `Crypt` (AES) للحقول الشخصية |
| الاختبارات | PHPUnit عبر `php artisan test` على قاعدة `kafaat_test` |

- **الخلفية** واجهة REST محضة تحت البادئة `/api` (`routes/api.php`)، بلا واجهات ويب مُخدَّمة من الخادم.
- كل المسارات المحمية تحت `auth:sanctum`؛ تسجيل الدخول والبوابة العامة مقيّدة بالمعدّل (throttle) ضدّ التخمين.

## 3. نموذج البيانات الأساسي

```
Sector (قطاع)
  └── Candidate (الشخص — بياناته الشخصية مشفّرة، له تصنيف أمني)
        └── Assessment (دورة تقييم — رمز فريد + حالة + رمز تأكيد)   [1 شخص ← عدّة دورات]
              ├── Schedule (موعد جلسة: تاريخ/وقت/نشاط/مكان)
              │     └── Attendance (حضور/غياب للجلسة — واحد لكل جلسة)
              ├── Evaluation (تقييم نشاط بواسطة مُقيّم — حالة مستقلّة)
              │     └── EvaluationScore (درجة كفاءة — فريدة داخل التقييم)
              └── FinalReport (تقرير نهائي واحد للدورة)

Competency (كفاءة) ── activity_competency ── (ربط النشاط بالكفاءات المطلوبة)
User (مستخدم النظام) ── Role (دور) ── Permissions (مصفوفة صلاحيات في الكود)
AuditLog · Notification · ChatThread/ChatMessage · SmsLog/EmailLog · Setting
```

**حقول الحالة (enums):**
- `candidate.status` و `assessment.status`: `draft → scheduled → assessed → approved → completed`
- `evaluation.status`: `draft → submitted → approved` (أو `returned`/إرجاع لـ`draft`)
- `report.status`: `draft → pending_dev_approval → approved` (أو `returned`)
- `classification`: `normal · secret · top_secret`
- `tier` (الفئة القيادية): `upper · middle` — تُحسب آلياً من الرتبة
- `assessment_type`: `comprehensive (شامل) · executive (تنفيذي)`

**ثابت جوهري:** كل مرشّح له دورة تقييم واحدة على الأقل، وحالة المرشّح تُزامَن دائماً مع دورته **الحالية (الأحدث)** عبر `Candidate::setStatus()`. الدورات المكتملة سجلٌّ تاريخي لا يُعاد كتابته.

## 4. دورة حياة المرشّح (آلة الحالة)

```
[إضافة/استيراد]  →  draft
      │  approve()            (اعتماد — SCHEDULER/ADMIN)
      ▼
   scheduled  →  [دعوة + جدولة + حضور]
      │  تقييم كامل ثمّ submit()  (المُقيّم)
      ▼
   assessed   →  [كتابة التقرير النهائي]
      │  إنشاء تقرير + إرساله (report.create) → pending_dev_approval
      │  اعتماد التقرير approve()  (DEV_MANAGER)
      ▼
   completed  →  [reassess() يفتح دورة جديدة draft لنفس الشخص]
```

انتقالات محروسة (بعد إصلاحات المراجعة):
- `approve()` لا يعمل إلا من `draft` (لا يُعيد دورة مكتملة للخلف).
- `returnEvaluation()` يُنزِل الحالة **فقط** على الدورة الحالية إن لم يبقَ لها تقييم مُرسل/معتمد آخر.
- اعتماد التقرير يُكمِل الدورة والمرشّح معاً (`completed`).

## 5. الخدمات (المتحكّمات) — كلٌّ ودوره

> كل متحكّم يمثّل «خدمة نطاق» في الـAPI. تحت كل خدمة: دورها، أبرز نقاطها، وقواعد العمل المميّزة.

### 5.1 المصادقة — `AuthController`
دور الدخول والخروج وإدارة كلمة المرور الذاتية.
- `POST /login` — يصادق **داخلياً عبر AD** أو **محلياً**؛ يقفل الحساب ١٥ دقيقة بعد ٥ محاولات فاشلة (عدّاد ذرّي)؛ انقطاع AD يُصنّف «غير متاح» فلا يُعاقب المستخدم (لا قفل، لا رجوع لكلمة مرور محلية).
- `GET /me` · `POST /logout` · `POST /change-password` — تغيير كلمة المرور المحلية يمسح علم «يجب التغيير».
- **قيد معدّل:** `throttle:10,1` ضدّ رشّ كلمات المرور. الرسائل عامّة (لا تكشف وجود/نوع الحساب).

### 5.2 المرشحون — `CandidateController`
الخدمة الأكبر: دورة حياة الشخص ودوراته.
- `GET /candidates` (فلترة بالحالة/القطاع/الفئة/التصنيف، بحث بالرمز) · `GET /candidates/{id}` · `GET /candidates/stats`.
- `POST /candidates` — إضافة/إعادة تقييم لشخص عبر **ديدَاب الهوية** (شخص واحد ← عدّة دورات)؛ يُنشئ دورة + رمز فريد + رمز تأكيد، ويرسل رسالة تأكيد. تعيين تصنيف أمني يتطلّب صلاحية `view_classified`.
- `PUT /candidates/{id}` — تعديل؛ يزامن نوع التقييم على الدورة **غير المكتملة** فقط (لا يمسّ التاريخ).
- `DELETE /candidates/{id}` — حذف مسودة/مجدول فقط، **بسبب إلزامي**، والحذف + التوثيق في معاملة ذرّية واحدة (خطّاف `deleting` ينظّف سجلّات الرسائل قبل الحذف).
- `POST /candidates/{id}/approve` — `draft → scheduled` (محروس).
- `POST /candidates/{id}/reassess` — دورة جديدة لمرشّح مكتمل (يُمنع مع دورة نشطة).
- `PATCH /candidates/{id}/classify` — تغيير التصنيف الأمني (حوكمة، مُدقَّق).
- `GET /candidates/{id}/assessments` — سجلّ الدورات وتقييماتها (للتاريخ/التقييم السابق).
- `GET /candidates/{id}/journey` — **خطّ زمني كامل** (إضافة → جدولة → حضور → تقييم → تقرير → اعتماد).
- `GET /candidates/export` — تصدير القائمة (يحترم التصنيف ورؤية الأسماء).
- **بوابة التصنيف:** أي مرشّح مصنّف خارج صلاحية المستخدم يُعامَل كـ«غير موجود» (٤٠٤).

### 5.3 الاستيراد الجماعي — `ImportController`
- `POST /candidates/import` — استيراد صفوف (هوية/اسم/جوال/بريد/كود القطاع/الرتبة)؛ يتحقّق سطراً سطراً، ويُنشئ **مرشّح + دورة** معاً في معاملة، بنفس مولّد الرموز الموحّد. يميّز تكرار الهوية عن تصادم الرمز المتزامن، ولا يُسرّب رقم الهوية في الرسائل.

### 5.4 التقييم — `EvaluationController`
دور إدخال درجات الكفاءات لكل نشاط واعتمادها.
- `GET /competencies` · `GET /evaluations` (تقييماتي، لاستكمال المسودات) · `GET /evaluations/{id}`.
- `POST /evaluations/start` — يبدأ جلسة لنشاط (`interview`/`discussion`) لمرشّح معتمد؛ فهرس فريد `(assessment_id, activity)` يمنع التكرار حتى تحت التزامن.
- `POST /evaluations/{id}/scores` — حفظ ذرّي للدرجات (استبدال كامل)، مع التحقّق من انتماء الكفاءة للنشاط والحدّ الأقصى.
- `POST /evaluations/{id}/submit` — يشترط رصد **كل** كفاءات النشاط؛ ينقل المرشّح `scheduled → assessed` ويُشعِر مدير التقييم.
- `POST /evaluations/{id}/approve` · `POST /evaluations/{id}/return` — اعتماد/إرجاع (صلاحية `evaluation.approve`)؛ الإرجاع يُنزِل الحالة على الدورة الحالية فقط.
- **سرّية الدرجات:** المُقيّم يرى تقييماته فقط؛ المراجعون يرون الكل.

### 5.5 الكفاءات والأنشطة — `ActivityCompetencyController`
- `GET /activity-competencies` · `PUT /activity-competencies/{activity}` — ربط كفاءات كل نشاط (بديل الـseeder الثابت). يمنع **حذف/تبديل** كفاءات نشاط له تقييمات نشطة (الإضافة فقط مسموحة) حتى لا تُكسَر التقييمات الجارية.

### 5.6 التقارير — `ReportController`
دور كتابة التقرير النهائي واعتماده.
- `GET /reports` (بحث بالهوية عبر الهاش) · `GET /reports/stats` · `GET /reports/eligible-candidates` (جاهزون بلا تقرير) · `GET /reports/{id}`.
- `POST /reports` — إنشاء (بعد اكتمال التقييم فقط)؛ فهرس فريد **تقرير واحد لكل دورة**.
- `PUT /reports/{id}` — تعديل · `POST /reports/{id}/approve` (يُكمِل المرشّح) · `POST /reports/{id}/return` · `POST /reports/{id}/resubmit`.
- **ملكية التحرير:** المؤلّف أو من يملك `report.edit_any` (مدير التقييم). درجات التوافق (`behavioralFit`/`technicalFit`) رقمية موحّدة عبر كل النقاط.

### 5.7 الحضور — `AttendanceController`
- `GET /attendance/today` · `GET /attendance/stats` — لوحة اليوم.
- `POST /attendance/{scheduleId}/checkin` · `POST /attendance/{scheduleId}/absence` — تسجيل حضور/غياب لجلسات اليوم فقط؛ **حضور واحد لكل جلسة** (فهرس فريد)، وتصادم التكرار المتزامن يُترجَم لـ٤٢٢ (بتمييز خطأ الفهرس 23505 عن غيره).

### 5.8 الاتصالات — `CommunicationController` (+ `CommunicationService`)
- `POST /communications/invite` — دعوة بالبريد و/أو الرسالة النصية؛ يُوثَّق فقط عند إرسال فعلي (مع نتيجة كل قناة).
- `GET /communications/history/{candidateId}` — سجلّ المراسلات؛ النص يُكشف لمن يملك رؤية الأسماء، و**رابط التأكيد يُحجب دائماً** (رمز حيّ)، مع فك تشفير محروس للصفوف القديمة.

### 5.9 البوابة العامة — `PublicAssessmentController`
بوابة المرشّح بلا مصادقة نظام، عبر رابط فريد في الرسالة.
- `POST /public/assessment/{token}/verify` — **العامل الثاني:** لا تُكشف أي بيانات إلا بمطابقة رقم الهوية (مقارنة ثابتة الزمن)؛ قفل تخمين بعد ٥ محاولات (زيادة ذرّية ضد TOCTOU)؛ يُصدر رمز جلسة **عديم الحالة** مشفّراً (٢٠ دقيقة).
- `POST /public/assessment/{token}/confirm` — تأكيد البيانات مرّة واحدة (idempotent).
- `POST /public/assessment/{token}/arrive` — تسجيل وصول ذاتي يُعلّم حضور جلسات اليوم عبر `insertOrIgnore` (آمن تحت التزامن، بلا 500).
- رابط غير صالح وهوية غير مطابقة يعطيان **نفس الردّ** (منع التعداد). قيد معدّل `throttle:20,1`.

### 5.10 الإشعارات — `NotificationController` (+ `NotificationService`)
- `GET /notifications` (ترقيم، الافتراضي 50) · `GET /notifications/unread-count` · `PATCH /notifications/{id}/read` (ملكية) · `PATCH /notifications/read-all`.
- الخدمة `NotificationService`: `notify()` لمستخدم و`notifyRole()` لكل من يحمل دوراً — تُستخدم في مسارات الاعتماد.

### 5.11 المحادثات — `ChatController`
نقاش مُلحق بكيان (تقرير) لتبادل الملاحظات.
- `GET /chat/{entityType}/{entityId}` — يجلب/يُنشئ محادثة الكيان (بعد التحقّق من صلاحية الكيان وتصنيفه).
- `POST /chat/{threadId}/message` — إرسال؛ **كل رسائل المستخدمين من نوع `comment`** (منع انتحال رسائل النظام/الإجراءات)؛ المحادثة المغلقة ترفض الإرسال.

### 5.12 المستخدمون والأدوار — `UserController`
- `GET /users` · `GET /users/roles` · `GET /users/role-permissions` · `POST /users` · `PUT /users/{id}` · `PATCH /users/{id}/toggle` · `PATCH /users/{id}/password`.
- **حرّاس الذات:** لا تُغيّر دورك، لا تعطّل حسابك. تغيير الدور/التعطيل/إعادة تعيين كلمة المرور **يُبطل جلسات** المستخدم فوراً. المستخدم الداخلي (AD) لا تُعاد له كلمة مرور محلية (يمنع حلقة حبس).

### 5.13 التدقيق — `AuditController`
- `GET /audit/log` — سجلّ النظام (فلترة بالإجراء/المستخدم/التاريخ)؛ يحجب تفاصيل المرشّحين المصنّفين عمّن لا يملك التصريح (فشل مغلق)، دون حجب الإجراءات الجماعية (تصدير/استيراد).
- `GET /candidates/{id}/history` — سجلّ مرشّح بعينه (يحترم التصنيف، فشل مغلق لمرشّح محذوف/مصنّف).

### 5.14 الإعدادات — `SettingsController`
- `GET/PUT /settings/ldap` — إعداد الدليل النشط (كتابة ذرّية للمفاتيح + التدقيق في معاملة واحدة).
- `POST /settings/ldap/test` — اختبار الاتصال؛ المنفذ محصور بمنافذ LDAP القياسية (389/636/3268/3269) لمنع استخدامه كماسح منافذ (SSRF)، وكل محاولة مُدقَّقة.

### 5.15 القطاعات — `SectorController`
- `GET /sectors` — قائمة القطاعات (تُستخدم في الإضافة والفلترة). *(القطاعات تُدار عبر البذور/seeders حالياً.)*

### خدمات الطبقة الداخلية (`app/Services`)
| الخدمة | الدور |
|---|---|
| `CommunicationService` | إرسال البريد/الرسائل النصية عبر مزوّد + تسجيلها (مع تشفير PII في السجلّ). |
| `ActiveDirectoryService` | مصادقة LDAP للمستخدمين الداخليين + اختبار الاتصال؛ يميّز «غير متاح» عن «اعتماد خاطئ» عبر رمز خطأ LDAP. |
| `NotificationService` | إنشاء الإشعارات لمستخدم/دور، مربوطة بالكيان والفاعل. |

> **ملاحظة:** الجدولة (`Schedule`) كيانٌ تستهلكه خدمات الحضور/الرحلة/البوابة العامة، لكن إنشاء الجداول ليس مكشوفاً كنقطة REST مستقلّة في السطح الحالي (تُبذَر بيانات العرض عبر الـseeders).

## 6. طبقة الأمان

### 6.1 التحكّم بالصلاحيات (RBAC)
- **مصفوفة صلاحيات في الكود** (`app/Security/Permissions.php`، 29 صلاحية) تربط كل **دور** بصلاحياته. `ADMIN` يملك `*`.
- الأدوار: `ADMIN · CENTER_MANAGER · SCHEDULER · RECEPTIONIST · ASSESS_MANAGER · EVALUATOR · DISCUSSION_EVAL · ASSISTANT · DEV_MANAGER · MEASURE_SUPER · EXTERNAL_ADD`.
- الصلاحيات تُرسَل للواجهة عند الدخول لضبط العرض، وتُفحَص خادمياً في كل نقطة.

### 6.2 التصنيف الأمني للبيانات
- ثلاثة مستويات: `normal / secret / top_secret`. من لا يملك `candidate.view_classified` يرى العاديين فقط، وأي مصنّف يظهر له كـ«غير موجود» (٤٠٤) — لا كشف وجود.
- رؤية الأسماء وأرقام الهوية محكومة بصلاحية منفصلة (`candidate.view_names`)؛ كل اطّلاع على PII يُدقَّق.

### 6.3 التشفير وحماية PII
- حقول المرشّح الحسّاسة (الاسم/الهوية/الجوال/البريد) مشفّرة بـ`Crypt` في القاعدة، مع `national_id_hash` (SHA-256) للبحث والديدَاب بلا فكّ تشفير.
- سجلّات الرسائل/البريد تُشفّر نصوصها وأرقامها (لا PII واضحة في النسخ الاحتياطية).

### 6.4 سجلّ التدقيق
- كل فعل حسّاس (إنشاء/تعديل/حذف/اطّلاع PII/تصدير/تصنيف/قفل حساب/اعتماد…) يُكتب مع الفاعل والوقت والـIP، ويُحترَم التصنيف عند العرض.

### 6.5 ضوابط أخرى
- **قفل الحساب** بعد ٥ محاولات (عدّاد ذرّي)، و**قيود معدّل** على الدخول والبوابة العامة.
- **بوابة عاملَين** للمرشّح: رابط تملكه + هوية تعرفها، برمز جلسة عديم الحالة.
- **حاجز SSRF** على اختبار LDAP، ومنع **انتحال رسائل** المحادثة، و**تقييد المعرّفات الرقمية** في المسارات (404 بدل 500).
- **نزاهة على مستوى القاعدة:** فهارس فريدة (هوية المرشّح، حضور الجلسة، تقييم النشاط، تقرير الدورة، درجات الكفاءة، محادثة الكيان).

## 7. الواجهة الأمامية (Vue)

- `src/views/` — صفحات: المرشحون، التقييم، التقارير، الحضور، المستخدمون، التدقيق، الإعدادات، وصفحة `Confirm` العامة للمرشّح.
- `src/services/api.js` — عملاء الـAPI (بما فيها `publicApi` للبوابة و`reportApi`)؛ معترِض يتعامل مع انتهاء الجلسة دون كسر صفحة `/confirm`.
- `src/services/perms.js` — فحص الصلاحيات في الواجهة لضبط العرض.
- `src/router/` — مسار `/confirm/:token` العام + حارس التوجيه؛ التركيب يُثبِّت الجذر قبل التوجيه (إصلاح إعادة التوجيه للدخول).
